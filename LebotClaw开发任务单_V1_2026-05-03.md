# LebotClaw — 面向 K-12 教育的 CLI 智能体框架

> 基于 OpenClaw 的工具调用与持久记忆架构 + Hermes 的长程规划、跨会话记忆与自学习能力，面向 K-12 教育场景深度改造的开源 CLI 智能体框架。
> 运行形态定位为可在本地电脑终端运行、也可在服务器上长期部署的教育智能体运行时，形态上更接近 OpenClaw / Hermes，而不是嵌入式端或具身硬件控制系统。

---

## 项目定位

LebotClaw 是一个面向 K-12 教育场景的 CLI 智能体框架，既可在本地电脑终端运行，也可部署在服务器上长期运行，核心解决两个问题：
1. **教育场景的智能体能力编排** — 将工具调用、长程规划、跨会话记忆、多轮交互封装为儿童可理解、教师可配置的操作单元
2. **长期可用的教育智能体体验** — 让智能体能够在本地或服务器环境中持续运行，记住学生画像、学习进度、错题轨迹和教学偏好，而不是每轮对话都“重新开始”

**与 OpenClaw 的关系**：不仅借鉴其 ToolCall 工具调用框架和 Agent 注册机制，也吸收其“长期运行智能体”思路，将会话状态、用户偏好、学习进度、工具结果摘要沉淀为可检索的持久记忆，形成“工具执行 -> 记忆写入 -> 后续调用时定向检索注入”的闭环，适配为教育场景下的学生画像与学习档案能力。

**与 Hermes 的关系**：不仅借鉴其长程任务分解与规划能力，也吸收其“记忆会成长、技能会沉淀”的核心思想，将复杂教学任务执行后的有效做法沉淀为可复用的教学技能模板、错题讲解模板、学习路径模板，并结合跨会话检索与用户建模，使智能体在持续使用中逐步形成更稳定的个性化教学策略。

**产品形态定位**：LebotClaw 不是一个面向硬件控制的嵌入式 Agent，也不是一个纯聊天机器人，而是一个可运行在本地终端、也可部署到服务器上的教育智能体 CLI Runtime。用户可以像使用 OpenClaw 或 Hermes 一样，在终端中与它持续对话、调用工具、切换学科 Agent、查看学习计划和复盘记录；团队也可以把它部署到云服务器或内网服务器上，作为长期运行的教育智能体服务。

## 设计原则

1. **CLI First**：优先保证终端内的可用性、可观测性和开发效率，先做强本地 CLI，再考虑扩展 GUI 或消息渠道。
2. **OpenClaw 式运行时**：强调工具调用、会话路由、Agent 注册、长时间运行和本地可控。
3. **OpenClaw 式记忆闭环**：不是只保留聊天历史，而是将学生偏好、知识盲点、任务上下文、学习目标结构化沉淀并按需召回。
4. **Hermes 式成长机制**：不是一次性回答，而是在规划、执行、复盘中沉淀技能和教学策略，让智能体“越用越懂学生”。
5. **教育优先**：所有能力最终服务于“分步引导、个性化学习、低延迟响应、可解释反馈”。

## 开发策略声明

这份任务单的作用是给出 **目标、优先级和推荐路径**，而不是把实现过程锁死成唯一方案。实际开发过程中，应坚持“以终为始、以最终可跑通的结果为导向”，而不是机械地逐条照抄任务顺序。

### 核心原则

1. **结果导向优先**：优先保证 MVP 跑通，先拿到可用链路，再逐步优化结构和抽象。
2. **问题驱动迭代**：开发过程中遇到工具调用、记忆检索、CLI 交互、会话持久化、模型适配等问题时，不要硬扛，应立即去查成熟开源实现。
3. **开源优先复用**：凡是社区已有成熟方案的部分，优先参考、复用、改写，而不是从零发明。
4. **允许偏离任务顺序**：如果实践中发现“先做路由”比“先做某个抽象类”更能跑通系统，可以调整顺序。
5. **以最终体验倒推实现**：始终从“最后用户怎么用 LebotClaw”反推现在该做什么，而不是被中间过程绑住。

### 实际执行建议

- 如果 `ToolCall` 协议迟迟跑不通，就优先去看 OpenClaw、OpenHands、agent-runner 一类项目的工具执行链路。
- 如果 CLI/TUI 体验不顺手，就优先去看 OpenCode、Crush 这类终端 Agent 项目的命令设计、会话恢复、配置组织方式。
- 如果规划器和状态流很乱，就优先看 Hermes、LangGraph 一类项目的状态机、图编排、持久化思路。
- 如果服务端部署和长期运行不稳定，就优先看 OpenClaw、OpenHands、open-agent-studio/agent 这类“本地 + 远端 + 常驻进程”方案。

### 关于“抄代码”和“借鉴代码”

本项目建议明确采用 **开源优先的工程策略**：
- 能直接复用的实现就复用
- 能局部改造的模块就改造
- 能通过阅读开源项目快速获得稳定方案的，就不要重复踩坑

但要注意两点：
- 优先参考 MIT、Apache-2.0 等许可友好的项目，并保留必要署名与 license 说明
- 不要无差别复制整套系统，而是要围绕 LebotClaw 的教育目标做“选择性借鉴 + 定向改造”

## 技术参考详解

这一部分不是简单写“参考了 OpenClaw 和 Hermes”，而是明确说明：我们究竟参考了它们哪些工程思想、运行机制、模块拆分方式，以及这些能力在 LebotClaw 里如何落地。这样同事在后续开发时，不会把“参考某项目”理解成一句空话。

### 一、对 OpenClaw 的技术参考

OpenClaw 值得参考的，不只是“能调工具”，而是它把一个长期运行 Agent 所需的核心部件组织成了完整运行时。对于 LebotClaw 来说，最重要的启发主要有 6 个层面：

#### 1. 运行时整体形态（Runtime Shape）

OpenClaw 不是一个单轮问答脚本，而是一个长期运行的本地 Agent Runtime。它把以下能力组织成一个统一系统：
- 会话入口
- Agent 执行循环
- 工具调用
- 会话/上下文状态
- 工作区文件
- 持久记忆
- 技能扩展

对 LebotClaw 的启发是：我们也不应该只做一个 `chat.py`，而应该从一开始就把它设计成一个“可持续运行的教育智能体运行时”。虽然 MVP 先落在 CLI 中，但内部结构要按照长期可扩展的 Agent Runtime 来搭。

#### 2. 工具调用协议（Tool Calling Protocol）

OpenClaw 的核心强项之一，是它把“模型输出 -> 识别工具调用 -> 执行工具 -> 把结果再喂回上下文”这件事做成了稳定的运行机制，而不是写死在某个 prompt 里的临时技巧。

LebotClaw 对应落地为：
- `Tool` 基类：所有教育工具统一接口
- `ToolRegistry`：统一注册与发现机制
- `ToolCall` 协议：从模型输出中解析工具调用意图
- Tool Result Injection：把工具结果注入当前上下文或记忆层

这对教育场景非常关键。因为很多学生请求并不需要“大模型自由发挥”，而是应先走工具链：
- 算术计算 -> 计算器工具
- 词语解释 -> 字典工具
- 知识点定位 -> 知识检索工具
- 学习计时 -> timer 工具

这样做的好处是：
- 降低模型调用成本
- 提高结果确定性
- 让系统更可测试
- 为后续路由策略提供明确边界

#### 3. 本地工作区与长期运行思维

OpenClaw 很重要的一点，是它默认智能体是“活着的”。它不是浏览器里一次性的对话框，而是有工作区、有文件、有持续状态、有历史轨迹的 Agent，并且天然适合长期运行在本地环境或服务器环境中。

LebotClaw 借鉴这一点后，产品定位也应该非常明确：
- 它是本地终端和服务器环境都可运行的教育助手
- 它有自己的工作区
- 它可以保存学习记录
- 它可以沉淀教学模板
- 它可以在多次使用后越来越了解某个学生

这意味着我们的目录设计、日志设计、记忆设计都不能只围绕“这轮对话”来写，而要围绕“这个学生长期使用这个教育 Agent”来设计。

#### 4. 记忆闭环而不是聊天历史堆积

OpenClaw 相关生态里最有价值的思想之一，是“记忆不是把所有聊天记录无脑塞进上下文”，而是把真正重要的内容沉淀下来，在需要时再精准召回。

对应到 LebotClaw，我们应把记忆分成更适合教育场景的几类：
- 学生画像记忆：年级、偏好、常见情绪、学习风格
- 学习进度记忆：学到哪一章、哪些知识点已掌握
- 错题/卡点记忆：经常出错的位置、易混概念
- 教学过程记忆：上一次讲解用了什么方法，效果如何
- 工具结果摘要记忆：某次检索、计算、规划的关键结论

真正需要实现的不是“保存全部历史”，而是以下闭环：
1. 对话或工具执行后抽取关键教育记忆
2. 将记忆持久化保存
3. 下次对话按当前意图和学科检索相关记忆
4. 将最相关记忆注入 prompt 或规划器
5. 再根据执行结果更新记忆

这才是 LebotClaw 里“参考 OpenClaw 记忆能力”的实质。

#### 5. Agent 注册与多 Agent 组织方式

OpenClaw 的另一个工程启发是，Agent 本身应是可注册、可路由、可切换的，而不是在一个大 prompt 里硬拼所有角色。

LebotClaw 对应做法：
- 数学学伴 Agent
- 语文助手 Agent
- 科学探究 Agent
- 通用教育调度 Agent

通过 `AgentRegistry` 和路由器统一管理。这样做的好处是：
- 每个学科可有独立工具集
- 每个学科可有独立 prompt
- 每个学科可有独立记忆策略
- 后续扩展英语、编程、心理陪伴更自然

#### 6. 安全与可控边界

OpenClaw 是本地 Agent，因此非常强调“能力强”与“边界清晰”并存。虽然 LebotClaw 是教育产品，不会一开始开放危险工具，但这套思想仍然应该继承：
- 工具白名单
- 不同 Agent 的工具权限隔离
- 对外部检索结果做摘要与过滤
- 对敏感内容和不适龄内容做规则拦截

对 K-12 场景来说，这部分不是附属能力，而是产品可信度的基础。

### 二、对 Hermes 的技术参考

Hermes 真正厉害的地方，也不只是“会规划”，而是它把“长期运行的 Agent 如何不断成长”做成了体系。对 LebotClaw 来说，最有价值的启发主要有 5 个层面：

#### 1. 长程规划（Planning）

Hermes 非常强调把复杂目标拆成可执行步骤，而不是直接给一个大而泛的答案。

LebotClaw 在教育场景下的对应方式非常自然：
- 复习某一章，不是直接输出总结，而是拆成“知识点回顾 -> 例题练习 -> 错题分析 -> 小结”
- 学习一个新概念，不是直接给定义，而是拆成“引入 -> 举例 -> 练习 -> 检查 -> 拓展”

这意味着规划器在 LebotClaw 中不是可选项，而是教学体验的核心模块。

#### 2. 记忆会成长（Memory That Grows）

Hermes 的一个关键理念是：Agent 不只是“记住”，而是会随着使用积累越来越有结构的认知。

LebotClaw 可以把这个理念翻译成：
- 越用越懂某个学生
- 越用越知道这个学生哪里容易卡住
- 越用越知道什么讲解方式更有效
- 越用越能预测下一步该用哪种教学策略

所以 LebotClaw 的记忆不应只做 KV 存储，而应至少具备以下方向：
- 按主题组织
- 按学科组织
- 按时间和最近活跃度组织
- 支持摘要、检索、更新、淘汰

#### 3. 技能沉淀（Skill Formation）

Hermes 很有代表性的思想是：当 Agent 成功做完一类复杂任务后，不应让经验白白消失，而应该把经验沉淀为可以复用的技能。

这一点放到 K-12 教育场景中，价值非常大。因为教学天然有大量可复用模式：
- 分数概念的讲解套路
- 小学应用题的引导模板
- 作文批改的反馈框架
- 科学实验探究的提问流程
- 情绪安抚与鼓励式反馈模板

LebotClaw 应把这些成功教学链路抽象为 `SkillLibrary` 中的可复用技能，而不是每次都完全从头生成。这样能显著提升：
- 稳定性
- 一致性
- 可解释性
- 复用效率

#### 4. 用户建模（User Modeling）

Hermes 强调 Agent 不只是处理任务，还会逐渐形成对使用者的理解。

LebotClaw 对应的“用户建模”就是“学生建模”，至少包括：
- 年龄 / 年级段
- 当前学习目标
- 掌握程度
- 注意力特点
- 偏好的讲解方式
- 容易受挫的知识点

这部分如果做好，LebotClaw 才能真正体现“教育属性”，而不只是把通用 Agent 换了个学科 prompt。

#### 5. 复盘与再规划（Review and Replan）

Hermes 的规划不是静态列表，而是会根据中途反馈进行调整。

LebotClaw 里要把这个能力直接变成教学机制：
- 学生答对了 -> 可以加快节奏
- 学生连续出错 -> 自动退回前置概念
- 学生表现出畏难情绪 -> 先降低题目难度并增加鼓励反馈
- 学生偏离目标 -> 重新规划当前学习路径

这使得 `Planner.replan()` 成为真正体现教学智能的关键部分。

### 三、LebotClaw 对两者的综合改造方式

如果用一句同事容易理解的话来概括：

**OpenClaw 更像是在教我们“怎么把 Agent 做成一个长期可运行的系统”，Hermes 更像是在教我们“怎么让这个系统越运行越聪明、越像一个会成长的助手”。**

因此 LebotClaw 的技术落点不是简单拼接，而是这样的组合：
- 用 OpenClaw 的方式组织运行时、工具系统、Agent 注册和记忆闭环
- 用 Hermes 的方式组织规划器、技能沉淀、跨会话成长和用户建模
- 用 K-12 教育的方式重写记忆分类、工具集、学科 Agent 和评估体系

### 四、对同事的直接实现建议

如果后续同事要开始编码，建议不要把“参考 OpenClaw / Hermes”理解成泛泛而谈，而是直接拆成以下工程落地项：

1. 先做 OpenClaw 风格的运行时骨架
- `Agent`
- `AgentRegistry`
- `Tool`
- `ToolRegistry`
- `MemoryStore`
- `Router`

2. 再做 Hermes 风格的成长模块
- `Planner`
- `SkillLibrary`
- `StudentProfile`
- `replan()`

3. 最后做教育场景封装
- 学科 Agent
- 教学提示词
- 教育工具
- 评估模块

这样拆，团队成员会更容易知道：哪些是运行时基础设施，哪些是教育场景能力，哪些是“成长型智能体”的差异化核心。

### 五、建议持续参考的其他开源项目

除了 OpenClaw 和 Hermes，本项目还建议持续参考其他成熟开源仓库。原因很简单：LebotClaw 要做的是一个“CLI + 长期运行 + 工具调用 + 记忆 + 规划”的综合体，单看一个项目很难把所有问题都解决掉。

建议把“看开源实现”当成开发流程的一部分，而不是卡住了才临时救火。下面这些项目都值得放进研发参考池：

#### 1. OpenCode / Crush

这类项目最值得借鉴的是终端智能体产品形态本身：
- CLI/TUI 交互体验
- 会话恢复与 session 管理
- 配置文件组织
- 本地运行与远程 attach 思路
- 多模型接入方式

如果团队后面要把 LebotClaw 做得更像一个真正可用的终端 Agent，而不是一个只有 `python main.py` 的脚本，那么 OpenCode / Crush 的产品化细节非常值得研究。

#### 2. OpenHands

OpenHands 更适合借鉴这些部分：
- 工具执行与沙箱隔离
- Agent SDK 的模块化抽象
- GUI / CLI / API 多入口共用同一 Agent Runtime 的方法
- 服务化部署和云端运行思路

虽然 OpenHands 偏代码 Agent，但它在“如何把 Agent 做成真正可部署的系统”这件事上很有参考价值。

#### 3. LangGraph

LangGraph 更适合借鉴这些问题：
- 如何做状态化编排
- 如何做长程任务分解
- 如何做可恢复执行
- 如何让规划器、工具节点、记忆节点之间关系更清晰

如果后续发现 `Planner`、`Router`、`MemoryStore` 三者耦合过高，就可以参考 LangGraph 的 graph/state 设计思路做重构。

#### 4. agent-runner / open-agent-studio/agent

这类项目更适合参考：
- 通用 Agent 骨架怎么快速搭起来
- 后台常驻 daemon 怎么组织
- 任务分解、执行、失败重试如何设计
- 如何把 CLI、任务、工具、持久化整合成一个最小可用系统

对 LebotClaw 的价值在于：它们能帮我们快速理解“一个可跑起来的 Agent Runtime 最少需要哪些部件”。

## 开发过程中的参考原则

在实际推进时，建议团队把下面这条当作共识：

> **不一定完全按照这份任务单的既定思路去做，而是要在开发过程中持续遇到问题、持续查找开源项目、持续借鉴成熟实现，并始终以最终目标能否跑通、能否交付、能否对教育场景真正有用为导向来确定策略和方法。**

具体来说：
- 某个模块如果 1 小时内没有明显进展，就立即转向开源调研，不要在原地硬磨
- 某个开源仓库已经把核心问题解决得很好，就优先复用其设计思路，必要时直接移植局部实现
- 若任务单中的模块划分与真实开发阻力不匹配，应允许随时调整模块边界和开发顺序
- 只要最终能更快、更稳地达到 MVP 目标，过程中的方法调整都是合理的
